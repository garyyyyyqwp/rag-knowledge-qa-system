第 11 课

RAG vs 纯 LLM 对比分析

通过对比实验理解 RAG 的价值，掌握项目中 /compare 端点的设计

01

为什么要对比？

RAG 的价值不能只靠理论论证，需要通过同题对比实验来直观感受。本项目专门设计了 /compare 端点，让相同的题目分别经过 RAG 管道和纯 LLM 处理，实时对比两者的回答差异。

🔍 直观感受 RAG 增强效果

通过并排对比，可以清楚看到 RAG 如何利用私有知识库中的信息给出更精准的回答，而纯 LLM 只能依赖训练数据中的通用知识。

🚨 理解「幻觉」问题的严重性

当问题涉及文档中的专有信息时，纯 LLM 可能会编造出看似合理但完全错误的答案。对比实验让「幻觉」问题暴露无遗。

🎯 学会判断何时需要 RAG

并非所有问题都需要 RAG。对于常识性问题，纯 LLM 可能回答得又快又好。通过对比，你能建立对「RAG 适用场景」的直觉判断。

📊 量化评估 RAG 质量

对比实验是评估 RAG 系统质量的基础方法。通过人工对比或自动评分，可以衡量检索准确性、答案忠实度等维度。

02

对比场景设计

为了全面评估 RAG 的效果，我们设计了三种典型的对比场景：

场景 1：知识库中有明确答案

预期：RAG 准确引用原文，纯 LLM 可能编造

例如：「张三的生日是什么时候？」
→ 文档中写道：「张三出生于1990年5月1日」

✅ RAG 回答：「根据文档[1]，张三的生日是1990年5月1日。」
❌ 纯 LLM 回答：「抱歉，我无法知道张三的个人信息...」或编造一个日期

场景 2：知识库中没有答案

预期：RAG 诚实说明，纯 LLM 按常识回答

例如：「李四的生日是什么时候？」
→ 文档中没有李四的信息

✅ RAG 回答：「现有资料中未找到关于李四生日的信息。」
⚠️ 纯 LLM 回答：「我不知道李四的生日，因为您没有提供相关信息。」

场景 3：知识库中部分相关

预期：RAG 给出有依据的回答，纯 LLM 给出通用回答

例如：「反向传播算法的原理是什么？」
→ 文档中有深度学习章节提到了反向传播

✅ RAG 回答：「根据资料[2]，反向传播算法由Rumelhart等人在1986年提出。该算法利用链式法则计算损失函数相对于每个权重的梯度...」
⚠️ 纯 LLM 回答：「反向传播算法是训练神经网络的核心方法...」（通用教科书回答，无具体引用）

03

对比维度

从多个维度系统对比 RAG 增强与纯 LLM 的表现：

对比维度RAG 增强纯 LLM
准确性 高 基于真实资料，回答有据可查 不稳定 可能编造事实，存在幻觉风险
可追溯性 强 每条信息可追溯到具体文档和 Chunk 无 无法追溯信息来源
时效性 灵活 取决于知识库更新时间，可实时更新 受限 受限于模型训练数据的截止日期
领域深度 可定制 专业知识库可覆盖任意垂直领域 通用 基于通用训练数据，深度有限
响应速度 略慢 多了 Embedding + 向量检索 + Prompt 构建 较快 直接调用 LLM，无额外步骤
成本 略高 Embedding API + LLM API 双重费用 较低 仅 LLM API 费用
数据安全 可控 可本地部署 Embedding 和 VectorStore 依赖外部 完全依赖 LLM 服务商
04

项目中的 compare 端点

/api/v1/qa/compare 端点同时调用 RAG 管道和纯 LLM，并通过 SSE 交错返回两个答案：

Python — app/routers/qa.py
from sse_starlette.sse import EventSourceResponse
from app.services.rag_pipeline import rag_pipeline
from app.services.llm import generate_bare_answer_stream
from app.services.streaming import rag_compare_sse

@router.post("/compare")
async def compare_rag_vs_bare(request: QuestionRequest):
    """
    对比端点 — 同时获取 RAG 增强和纯 LLM 回答

    事件序列：
    rag_answer ←→ bare_answer (交错) → rag_citations → done

    两种回答的 token 交替发送，实现实时对比效果。
    """
    # 1. 执行 RAG 管道
    retrieved_chunks, rag_stream, citations = await rag_pipeline(
        question=request.question,
        top_k=request.top_k
    )

    # 2. 执行纯 LLM 生成（无上下文）
    bare_stream = generate_bare_answer_stream(request.question)

    # 3. 交错输出两种回答
    return EventSourceResponse(
        rag_compare_sse(rag_stream, bare_stream, citations)
    )

05

generate_bare_answer_stream 详解

纯 LLM 回答的生成函数与 RAG 回答的最大区别在于：没有任何上下文、没有 Prompt 模板、没有角色约束，只有用户的原始问题：

Python — app/services/llm.py
async def generate_bare_answer_stream(question: str) -> AsyncIterator[str]:
    """
    纯 LLM 回答生成 — 无任何 RAG 增强

    与 generate_answer_stream() 的关键区别：
    - 没有上下文注入（无 context）
    - 没有 Prompt 模板（无角色定义、无格式要求）
    - 只有用户的原始问题

    Args:
        question: 用户原始问题

    Yields:
        每个 token 作为单独的字符串
    """
    client = get_client()

    response = await client.chat.completions.create(
        model=get_model(),
        messages=[
            {"role": "user", "content": question}  # 只有问题，无上下文！
        ],
        stream=True,
    )

    async for chunk in response:
        delta = chunk.choices[0].delta
        if delta.content is not None:
            yield delta.content
关键区别总结：
RAG 回答 纯 LLM 回答
System Prompt✅ 角色定义❌ 无
参考资料✅ 检索到的 Chunks❌ 无
引用来源✅ 可追溯❌ 不可追溯
幻觉风险🟢 低🔴 高

对比实验示例

以下是用真实问题测试的结果对比：

RAG 增强回答

根据文档[1]，张三的生日是1990年5月1日，他在北京工作。文档还提到他于2015年加入公司，目前担任高级工程师。

根据文档[2]，他参与的核心项目包括智能客服系统和推荐引擎重构。

纯 LLM 回答

抱歉，我无法得知张三的个人信息。张三是一个很常见的名字，如果您能提供更多上下文（例如所在公司、职位等），我也许能给出更具体的回答。

作为AI助手，我没有访问个人数据的权限，也无法查询特定个人的信息。

RAG 增强回答（技术类问题）

根据资料[3]，反向传播算法由Rumelhart、Hinton和Williams在1986年的论文中正式提出。

该算法利用链式法则计算损失函数相对于网络中每个权重的梯度，从输出层向输入层逐层传播误差信号。资料[4]指出，反向传播使深层网络的训练成为可能，是深度学习的基础算法之一。

纯 LLM 回答（技术类问题）

反向传播算法是训练人工神经网络的核心方法。它通过计算损失函数对权重的梯度，利用优化算法（如SGD）更新权重。

该方法由多位研究者共同发展，是现代深度学习的基础。反向传播使用链式法则高效计算梯度，使深层网络的训练在计算上变得可行。

同步对比端点

除了流式的 /compare 端点，项目还提供了 /compare/sync 端点，返回完整的 JSON 响应，适合自动化测试和批量评估：

Python — app/routers/qa.py
class CompareResponse(BaseModel):
    """同步对比响应"""
    rag_answer: str
    bare_answer: str
    rag_citations: list[CitationInfo]

@router.post("/compare/sync", response_model=CompareResponse)
async def compare_sync(request: QuestionRequest):
    """
    同步对比端点 — 返回完整的 JSON 响应

    与 /compare 的区别：
    - 非流式，等待两个答案全部生成后一起返回
    - 适合测试脚本和批量评估场景
    """
    # 并行获取两个答案（非流式）
    rag_answer = await generate_full_rag_answer(request.question, request.top_k)
    bare_answer = await generate_full_bare_answer(request.question)
    citations = await get_citations_for_question(request.question, request.top_k)

    return CompareResponse(
        rag_answer=rag_answer,
        bare_answer=bare_answer,
        rag_citations=citations
    )
使用场景：/compare/sync 非常适合集成到测试脚本中。可以准备一组测试问题，批量调用此端点，用程序自动对比 RAG 和纯 LLM 的回答质量。

📋

本课总结

  • 通过 RAG vs 纯 LLM 对比实验可以直观感受 RAG 的价值和纯 LLM 的局限性
  • 设计了 3 种对比场景：知识库有答案、无答案、部分相关
  • 从 7 个维度对比：准确性、可追溯性、时效性、领域深度、速度、成本、安全
  • /compare 端点通过 SSE 交错发送 RAG 和纯 LLM 的答案 token
  • generate_bare_answer_stream() 无上下文、无 Prompt 模板，只有原始问题
  • RAG 的核心优势：准确、可追溯、时效灵活、深度可定制
  • /compare/sync 同步端点为批量测试和自动评估提供了便利
