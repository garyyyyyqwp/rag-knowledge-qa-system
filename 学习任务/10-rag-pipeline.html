02

Step 1: 检索 (Retrieve)

检索是 RAG 管道的第一步，也是决定答案质量的最关键步骤。如果检索不到正确的 Chunk，再好的 LLM 也只能产生无意义的回答。

Python — app/services/rag_pipeline.py
from app.schemas.qa import RetrievedChunkPreview
from app.services.vector_store import get_vector_store

async def retrieve_chunks(
    question: str,
    top_k: int = 5,
    store = None
) -> list:
    """
    将用户问题转为向量，在 ChromaDB 中检索最相关的 Chunks

    Args:
        question: 用户原始问题
        top_k: 返回的最相关结果数量
        store: ChromaDB 存储实例（可选，用于依赖注入）

    Returns:
        list[RetrievedChunk]: 按相似度降序排列的检索结果
    """
    store = store or get_vector_store()
    results = await store.search(query=question, top_k=top_k)

    chunks = []
    for r in results:
        chunk = {
            "doc_id": r["doc_id"],
            "filename": r["filename"],
            "chunk_index": r["chunk_index"],
            "content": r["content"],
            "content_preview": r["content_preview"],
            "score": r["score"],
        }
        chunks.append(chunk)

    return chunks
关键要点：
  • store.search(query, top_k) 内部自动调用 embed_single(query) 将问题转为向量
  • score 是余弦相似度，值越接近 1 表示越相关
  • top_k 的值很重要：太小可能遗漏相关信息，太大可能引入噪声干扰 LLM
  • 排名靠前的 Chunk 会出现在 Prompt 最前面，LLM 倾向于更关注前面的内容
03

Step 2: 构建上下文 (Build Context)

检索到的 Chunks 是散乱的文本片段，需要将它们格式化为结构清晰的上下文，方便 LLM 理解。同时要标注来源，让 LLM 能在回答中引用。

Python — app/services/rag_pipeline.py
def _build_context(chunks: list) -> str:
    """
    将检索到的 Chunks 格式化为上下文文本

    每个 Chunk 被编号并标注来源，便于 LLM 在回答中引用

    Args:
        chunks: retrieve_chunks() 的返回结果

    Returns:
        格式化后的上下文字符串
    """
    parts = []
    for i, chunk in enumerate(chunks):
        parts.append(
            f"[{i + 1}] (来源: {chunk['filename']})\n"
            f"{chunk['content']}"
        )
    return "\n\n".join(parts)

Step 3: Prompt 模板

Prompt 模板是 RAG 系统与 LLM 的 "接口契约"。好的 Prompt 设计决定了 LLM 能否正确使用检索到的资料、能否给出准确的回答。

Python — RAG Prompt 模板
RAG_PROMPT_TEMPLATE = """你是一个知识库问答助手。基于以下参考资料回答用户问题。

参考资料：
{context}

用户问题：{question}

要求：
1. 回答准确、简洁，直接回应用户的问题
2. 引用资料内容时标注来源编号，如 [1]、[2]
3. 如果资料不足以回答用户的问题，请明确说明"现有资料中未找到相关信息"
4. 不要编造资料中没有的内容"""
Prompt 设计的四个核心要素：
  1. 角色定义：明确 AI 的身份（知识库问答助手），约束行为边界
  2. 参考资料：将检索到的上下文注入 Prompt，这是 RAG 的关键
  3. 用户问题：保留原始问题，确保 LLM 明确知道要回答什么
  4. 格式要求：规定回答格式，包括引用标注、准确性要求、边界声明
Prompt 最佳实践：要求「如果资料不足以回答，请明确说明」非常重要。这可以大幅降低 LLM 的幻觉——如果文档中没有相关信息，它会诚实说明，而不是凭空编造。
05

Step 4: 流式生成 (Generate)

这是管道的最后一步——将组装好的 Prompt 发送给 LLM，并以流式方式逐 token 返回。注意这里使用的是 无历史消息的对话，只发送一条 user message。

Python — app/services/rag_pipeline.py
from app.services.llm import get_client, get_model
from typing import AsyncIterator

async def generate_answer_stream(
    question: str,
    chunks: list,
    template: str = RAG_PROMPT_TEMPLATE
) -> AsyncIterator[str]:
    """
    构建 Prompt 并流式生成 RAG 增强回答

    Args:
        question: 用户原始问题
        chunks: 检索到的相关 Chunks 列表
        template: Prompt 模板

    Yields:
        每个 token 作为单独的字符串
    """
    client = get_client()
    context = _build_context(chunks)
    prompt = template.format(context=context, question=question)

    response = await client.chat.completions.create(
        model=get_model(),
        messages=[
            {"role": "user", "content": prompt}
        ],
        stream=True,  # 开启流式输出
    )

    async for chunk in response:
        delta = chunk.choices[0].delta
        if delta.content is not None:
            yield delta.content
注意：
  • 使用的是 messages=[{"role": "user", "content": prompt}]，没有 system message
  • stream=True 返回的是异步迭代器，不是完整的响应
  • delta.content 可能为 None（某些 chunk 只包含元数据），需要检查
  • 每个 yield 对应一个 LangChain token，前端实时追加显示

引用提取

引用来源是 RAG 系统的关键特性——不仅给出答案，还能告诉用户答案来自哪里。这段代码在答案生成完毕后，从检索到的 Chunks 中提取引用信息：

Python — app/services/rag_pipeline.py
def _extract_citations(chunks: list) -> list:
    """
    从检索到的 Chunks 提取引用来源信息

    每个引用包含文档标识和内容摘要（前300字），
    方便前端展示和使用者判断来源可信度。

    Args:
        chunks: 检索到的 Chunks 列表

    Returns:
        list[Citation]: 引用信息列表
    """
    citations = []
    for c in chunks:
        citations.append({
            "doc_id": c["doc_id"],
            "filename": c["filename"],
            "chunk_index": c["chunk_index"],
            "content_snippet": c["content"][:300],
        })
    return citations
07

完整管道函数

最后，将检索、生成和引用提取三部分组装成一个完整的管道函数：

Python — app/services/rag_pipeline.py
async def rag_pipeline(
    question: str,
    top_k: int = 5,
    store = None
):
    """
    RAG 完整管道 — 一次调用完成检索 + 生成 + 引用提取

    这是整个 RAG 系统的核心入口函数。
    路由器 (routers/qa.py) 直接调用此函数获取所有结果。

    Args:
        question: 用户问题
        top_k: 检索结果数量 (默认5)
        store: ChromaDB 存储实例（可选，用于测试时注入 mock）

    Returns:
        tuple: (chunks, answer_stream, citations)
        - chunks: 检索到的相关文档片段
        - answer_stream: LLM 流式生成的答案 token 流
        - citations: 引用来源列表
    """
    # Step 1: 检索
    chunks = await retrieve_chunks(question, top_k, store)

    # Step 2: 流式生成
    answer_stream = generate_answer_stream(question, chunks)

    # Step 3: 引用提取
    citations = _extract_citations(chunks)

    return chunks, answer_stream, citations
这就是整个 RAG 系统的核心！三个步骤，十几行代码，但每一步都建立在前面所有知识的基础上：向量 Embedding → ChromaDB 存储与检索 → Prompt 工程 → LLM API 调用 → 流式输出。理解这个管道函数，就理解了 RAG 的全部。

本课总结

  • RAG 管道由 4 个核心步骤组成：检索 → 构建上下文 → Prompt 填充 → 流式生成
  • retrieve_chunks() 将问题 Embedding 后在 ChromaDB 中检索 Top-K 相关 Chunks
  • _build_context() 将 Chunks 格式化为编号的上下文字符串，标注来源文件名
  • RAG_PROMPT_TEMPLATE 包含角色定义、参考资料、问题、格式要求（4 要素）
  • generate_answer_stream() 使用 stream=True 逐 token 返回 LLM 响应
  • _extract_citations() 从检索结果中提取引用来源，支持答案可追溯
  • rag_pipeline() 是管道的统一入口，一行调用返回全部三个结果
