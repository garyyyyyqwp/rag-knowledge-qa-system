模块依赖关系

系统中有 8 个核心服务模块，每个模块有明确的职责和依赖关系：

服务文件职责依赖
文档解析 parser.py PDF (pdfplumber)、Markdown、TXT → 纯文本 无外部依赖
混合分块 chunker.py 文本 → List[Chunk]（基于 tiktoken 的智能分块） tiktoken
LLM 客户端 llm.py AsyncOpenAI 封装 — 流式/非流式调用 config.py
Embedding embedding.py 文本 → 1536 维向量 — 批量 + 单个 llm.py, config.py
向量存储 vector_store.py ChromaDB CRUD — 添加、搜索、删除文档 embedding.py
RAG 管道 rag_pipeline.py 编排检索→构建上下文→流式生成 embedding.py, vector_store.py, llm.py
流式输出 streaming.py SSE 事件格式化 — retrieval/answer/citations/done rag_pipeline.py
配置管理 config.py 环境变量读取 — OpenAI Key, Model, 路径 python-dotenv
依赖原则：依赖关系是单向的，没有循环依赖。config.py 是最底层的基础模块，streaming.py 是最上层的编排模块。Router 层不直接依赖外部 API，所有外部调用通过 Service 层进行。
03

数据流完整追踪

跟踪系统中最关键的两个数据流：文档上传索引 和 RAG 问答。

流程 1：文档上传索引

  • 步骤 1 POST /api/v1/documents/upload — 前端发送文件 (multipart/form-data)
    → routers/documents.py 接收 UploadFile
  • 步骤 2 parse_document() — 根据文件扩展名选择解析器
    → services/parser.py: PDF→pdfplumber, MD→直接读取, TXT→直接读取
  • 步骤 3 chunk_text() — 将纯文本切分为语义片段
    → services/chunker.py: RecursiveCharacterTextSplitter, chunk_size=500, overlap=50
  • 步骤 4 embed_texts() — 将每个 Chunk 转为 1536 维向量
    → services/embedding.py → OpenAI text-embedding-3-small API
  • 步骤 5 store.add_document() — 将 Chunks + Embeddings 存入 ChromaDB
    → services/vector_store.py: 元数据包含 doc_id, filename, chunk_index
  • 步骤 6 返回 DocumentUploadResponse
    → {doc_id, filename, file_type, chunk_count, status: "indexed"}

流程 2：RAG 问答

  • 步骤 1 POST /api/v1/qa/ask — 用户发送问题
    → routers/qa.py: 接收 QuestionRequest {question, top_k}
  • 步骤 2 rag_pipeline() — 管道入口
    → services/rag_pipeline.py
  • 步骤 3 retrieve_chunks() → store.search()
    → embed_single(question) → ChromaDB.query() → Top-K chunks with scores
  • 步骤 4 _build_context() — 格式化 Chunks
    → "[1] (来源: xxx.pdf)\n内容..." → 组装上下文
  • 步骤 5 generate_answer_stream() — Prompt + LLM
    → RAG_PROMPT_TEMPLATE.format() → AsyncOpenAI.chat.completions.create(stream=True)
  • 步骤 6 rag_ask_sse() — SSE 事件格式化
    → services/streaming.py: retrieval → answer (×N) → citations → done
  • 步骤 7 EventSourceResponse — 返回 SSE 流
    → Content-Type: text/event-stream → 前端 EventSource 消费
04

技术选型总结

回顾整个项目的技术栈，每个选择都有明确的原因：

层次技术选型理由
Web 框架 FastAPI 异步原生支持、自动生成 OpenAPI 文档、Pydantic 无缝集成、性能优异
LLM OpenAI GPT-4o-mini 性价比极高、API 稳定可靠、Streaming 支持好、中文能力强
Embedding text-embedding-3-small 1536 维向量、语义捕获能力强、价格低廉 (每 1000 token 约 ¥0.00014)、批量支持
向量数据库 ChromaDB 零配置、Python 原生、嵌入式运行、支持元数据过滤、持久化存储
文档解析 pdfplumber PDF 文本提取精度高于 PyPDF2、支持表格检测、中文兼容性好
Token 计数 tiktoken OpenAI 官方库、与模型 tokenizer 完全一致、计数准确
流式传输 sse-starlette FastAPI 原生 SSE 支持、异步生成器自动转换、轻量无依赖
测试框架 pytest + httpx 异步测试支持、强大的 Fixture 机制、httpx 原生 ASGI 测试客户端

# 服务配置

PORT=8000
Python — utils/config.py
import os
from dotenv import load_dotenv

load_dotenv()

def get_env(key: str, default: str | None = None) -> str:
    """获取环境变量，缺失时抛出明确错误"""
    value = os.getenv(key, default)
    if value is None:
        raise ValueError(f"缺少环境变量: {key}")
    return value

# 集中导出的配置对象
OPENAI_API_KEY = get_env("OPENAI_API_KEY")
LLM_MODEL = get_env("LLM_MODEL", "gpt-4o-mini")
EMBEDDING_MODEL = get_env("EMBEDDING_MODEL", "text-embedding-3-small")
CHROMA_PERSIST_DIR = get_env("CHROMA_PERSIST_DIR", "./chroma_data")
DEFAULT_TOP_K = int(get_env("DEFAULT_TOP_K", "5"))
CHUNK_SIZE = int(get_env("CHUNK_SIZE", "500"))
CHUNK_OVERLAP = int(get_env("CHUNK_OVERLAP", "50"))

扩展方向

当前系统已经是一个功能完整的 RAG 应用，以下是可能的扩展方向：

💬 多轮对话

添加对话历史记忆 — 将历史问答存入 ChromaDB 或外部数据库。在 Prompt 中注入最近的对话摘要，实现有上下文的连续对话。

🖼️ 图片 OCR

支持扫描件 PDF — 使用 Tesseract OCR 或云 OCR API 识别图片中的文字，再输入分块管道。大幅扩展可处理的文档类型。

👥 多租户

用户隔离 — 每个用户拥有独立的 ChromaDB collection，API 添加认证中间件。支持多用户各自管理知识库，互不干扰。

🐳 Docker 部署

容器化 — 编写 Dockerfile 和 docker-compose.yml，将 FastAPI、ChromaDB 分别容器化。简化部署流程，确保环境一致性。

🎨 前端 UI

React / Vue 界面 — 搭建可视化的文档上传、问答交互、引用展示界面。SSE 事件驱动前端实时渲染。

📄 更多文档格式

支持 .docx (python-docx)、.pptx (python-pptx)、.html (BeautifulSoup)、.epub 等格式，打造全能文档处理系统。
