SSE 流式输出原理与实践

理解 Server-Sent Events 协议，掌握项目中的结构化 SSE 事件流

01

什么是 SSE？

SSE (Server-Sent Events) 是一种基于 HTTP 的服务器向浏览器单向推送事件流的协议。它是 HTML5 标准的一部分，被所有现代浏览器原生支持。

SSE (Server-Sent Events)

  • 单向：服务器 → 客户端
  • 基于 HTTP 协议，无需额外握手
  • 轻量级，代码简单
  • 浏览器原生 EventSource API
  • 自动重连机制
  • 内容类型：text/event-stream

WebSocket

  • 双向：服务器 ↔ 客户端
  • 需要 HTTP → WebSocket 协议升级
  • 较重量级，需额外库支持
  • 需要手动管理连接状态
  • 需要自行实现重连逻辑
  • 内容类型：application/octet-stream

SSE 事件格式

SSE 使用简单的文本格式传输事件。每条消息由一个或多个字段组成，以空行结尾：

SSE 事件格式
# 一条完整的事件（以空行分隔）
event: answer
data: Transformer架构由Vaswani等人

# 下一条事件
event: answer
data: 在2017年提出，彻底改变了

# 最后一条事件
event: done
data: {}

核心规则：

  • event: 定义事件名称（可选，省略则触发 onmessage）
  • data: 事件数据（必需，可多行）
  • id: 事件 ID（可选，用于断线重连时定位）
  • 每条消息以空行 (\n\n) 结束
02

为什么 RAG 需要流式输出？

大语言模型 (LLM) 生成文本的方式是 token by token（逐 token 生成）。如果等所有 token 生成完毕再一次性返回给用户，会带来严重的问题：

⏳ 用户体验差

对于几百字的回答，LLM 可能需要 5-15 秒才能完整生成。用户看到白屏等待，体验极差。流式输出让用户看到答案逐字出现，如同对话，体验自然流畅。

📦 内存压力

非流式模式下，服务端必须在内存中累积完整的响应文本才能发送。对于长回答，这会占用大量内存。流式模式中每个 token 即时发送并释放。

🔗 超时风险

如果 LLM 生成时间超过 HTTP 连接超时（通常 30-60 秒），连接会被中断。SSE 通过持续发送数据保持连接活跃，避免超时。

📊 渐进式信息

本项目中，SSE 不仅流式传输答案文本，还流式传输检索结果和引用来源。用户可以在看到答案之前，先看到系统找到了哪些资料。

03

项目 SSE 事件结构

本项目定义了 4 种结构化 SSE 事件，每种事件有特定的语义和数据结构。它们按顺序发送，形成一条完整的信息流：

核心的 SSE 事件生成器代码：

Python — app/services/streaming.py
import json
from typing import AsyncIterator

async def rag_ask_sse(
    retrieved_chunks: list,
    answer_stream: AsyncIterator[str],
    citations_list: list
) -> AsyncIterator[dict]:
    """生成 RAG 问答的完整 SSE 事件流"""

    # 事件1：发送检索结果预览
    chunks_preview = [
        {
            "doc_id": c.doc_id,
            "filename": c.filename,
            "chunk_index": c.chunk_index,
            "content_preview": c.content[:200],
            "score": round(c.score, 4),
        }
        for c in retrieved_chunks
    ]
    yield {"event": "retrieval", "data": json.dumps({"chunks": chunks_preview})}

    # 事件2：逐 token 发送 LLM 生成的答案
    async for token in answer_stream:
        yield {"event": "answer", "data": token}

    # 事件3：发送引用来源列表
    citations = [
        {
            "doc_id": c.doc_id,
            "filename": c.filename,
            "chunk_index": c.chunk_index,
            "content_snippet": c.content[:300],
        }
        for c in citations_list
    ]
    yield {"event": "citations", "data": json.dumps(citations)}

    # 事件4：流结束信号
    yield {"event": "done", "data": "{}"}
04

sse-starlette 集成

FastAPI 本身不直接支持 SSE，但 sse-starlette 库提供了 EventSourceResponse 类，可以将异步生成器转换为标准的 SSE 响应：

Python — app/routers/qa.py
from sse_starlette.sse import EventSourceResponse
from app.services.rag_pipeline import rag_pipeline
from app.services.streaming import rag_ask_sse
from app.schemas.qa import QuestionRequest

@router.post("/ask")
async def ask_question(request: QuestionRequest):
    """
    RAG 问答端点 — 返回 SSE 流式响应

    事件序列：retrieval → answer (×N) → citations → done
    """
    # 1. 执行 RAG 管道 — 检索 + 生成
    retrieved_chunks, answer_stream, citations = await rag_pipeline(
        question=request.question,
        top_k=request.top_k
    )

    # 2. 包装为 SSE 事件流
    return EventSourceResponse(
        rag_ask_sse(retrieved_chunks, answer_stream, citations)
    )
关键点：
  • EventSourceResponse 自动设置 Content-Type: text/event-stream
  • 生成器的每个 yield 都会被即时发送给客户端（无需等待生成器结束）
  • 客户端断开连接时，生成器会被自动取消
  • 支持 Cache-Control: no-cache 防止中间代理缓存

对比端点：交错输出

/compare 端点是本项目的一大亮点。它同时获取 RAG 增强回答和纯 LLM 回答，并将两个答案的 token交错发送，让用户可以实时对比两种回答的差异：

Python — app/services/streaming.py
async def rag_compare_sse(
    rag_stream: AsyncIterator[str],
    bare_stream: AsyncIterator[str],
    citations_list: list
) -> AsyncIterator[dict]:
    """
    交错输出 RAG 回答和纯 LLM 回答

    事件序列：
    rag_answer ←→ bare_answer (交错) → rag_citations → done
    """
    rag_done = False
    bare_done = False
    rag_iter = rag_stream.__aiter__()
    bare_iter = bare_stream.__aiter__()

    while not (rag_done and bare_done):
        # 交替获取两个流的 token
        if not rag_done:
            try:
                rag_token = await rag_iter.__anext__()
                yield {"event": "rag_answer", "data": rag_token}
            except StopAsyncIteration:
                rag_done = True

        if not bare_done:
            try:
                bare_token = await bare_iter.__anext__()
                yield {"event": "bare_answer", "data": bare_token}
            except StopAsyncIteration:
                bare_done = True

    # 发送 RAG 的引用来源
    citations = [
        {"doc_id": c.doc_id, "filename": c.filename,
         "content_snippet": c.content[:300]}
        for c in citations_list
    ]
    yield {"event": "rag_citations", "data": json.dumps(citations)}
    yield {"event": "done", "data": "{}"}
注意：rag_answer 和 bare_answer 是交替发送的。这意味着前端可以同时渲染两列文本，让用户实时看到 RAG 是否比纯 LLM 给出更准确的回答。这是评估 RAG 价值的最直观方式。

前端消费 SSE

浏览器原生提供 EventSource 对象来消费 SSE 流。以下是在前端页面中消费本项目 SSE 事件的完整代码：

JavaScript — 前端
// 创建 SSE 连接
const eventSource = new EventSource('/api/v1/qa/ask');

// 监听检索结果事件
eventSource.addEventListener('retrieval', (e) => {
    const data = JSON.parse(e.data);
    console.log(`找到 ${data.chunks.length} 个相关片段`);
    renderRetrievalResults(data.chunks);
});

// 监听答案 token 事件（逐 token 追加）
const answerOutput = document.getElementById('answer-output');
eventSource.addEventListener('answer', (e) => {
    answerOutput.textContent += e.data;
    // 自动滚动到底部
    answerOutput.scrollTop = answerOutput.scrollHeight;
});

// 监听引用来源事件
eventSource.addEventListener('citations', (e) => {
    const data = JSON.parse(e.data);
    renderCitations(data);  // 渲染为可点击的链接列表
});

// 监听流结束事件
eventSource.addEventListener('done', (e) => {
    console.log('SSE 流结束');
    eventSource.close();  // 关闭连接
});

// 错误处理
eventSource.onerror = (e) => {
    console.error('SSE 连接出错:', e);
    if (eventSource.readyState === EventSource.CLOSED) {
        showError('连接已断开');
    }
};

EventSource 状态说明

状态值含义
CONNECTING0正在建立连接
OPEN1连接已建立，正在接收事件
CLOSED2连接已关闭（主动或被动）
最佳实践：
  • 在 done 事件中调用 eventSource.close() 释放连接
  • 处理 onerror — EventSource 会自动重连，但你可能需要处理永久性错误
  • 对于 /compare 端点，同时监听 rag_answer 和 bare_answer 事件

📋

本课总结

  • SSE 是服务器向浏览器单向推送事件流的 HTTP 协议，比 WebSocket 更轻量
  • SSE 事件格式：event: 事件名\ndata: 数据\n\n，每条消息以空行分隔
  • RAG 需要流式输出因为 LLM 是 逐 token 生成的，流式能极大提升用户体验
  • 本项目定义了 4 种结构化事件：retrieval、answer、citations、done
  • sse-starlette 的 EventSourceResponse 将异步生成器自动转换为 SSE 响应
  • /compare 端点实现了 RAG 与纯 LLM 回答的交错流式输出
  • 前端使用浏览器原生 EventSource API 消费 SSE，按事件类型分发处理
