FastAPI 微服务架构设计

理解 FastAPI 的三层架构（Router → Schema → Service），掌握项目整体结构

01

为什么选择 FastAPI？

在构建 RAG 系统时，我们需要一个 Web 框架来提供 API 接口。FastAPI 是 Python 生态中现代、高性能、开发效率极高的选择。以下是选择 FastAPI 的核心原因：

📖 自动生成 OpenAPI 文档

只需定义路由和 Pydantic Schema，FastAPI 自动生成符合 OpenAPI 3.0 规范的交互式文档。访问 /docs (Swagger UI) 或 /redoc (ReDoc) 即可查看和测试所有 API 端点。无需手写文档，文档即代码。

⚡ 异步支持 (async/await)

FastAPI 完全支持 Python async/await 语法，所有路由处理函数和依赖注入都可以是异步的。对于 LLM API 调用、Embedding 请求、数据库操作等 IO 密集型任务，异步能大幅提升并发处理能力。

✅ Pydantic v2 自动验证

Pydantic 提供强类型的数据定义和自动验证。请求参数、查询字符串、路径参数、请求体都会被自动校验。格式错误的请求会得到清晰的 422 错误响应，减少了大量手写验证代码。

🚀 高性能 (Starlette + Uvicorn)

FastAPI 底层基于 Starlette（轻量 ASGI 框架）和 Uvicorn（高性能 ASGI 服务器）。在性能基准测试中，FastAPI 的性能可与 Node.js 和 Go 的 Web 框架相媲美，远优于传统的 Flask + Gunicorn 组合。

02

三层架构设计

本项目采用清晰的三层架构来组织代码：Router 路由层 → Schema 数据层 → Service 业务层。各层职责明确，互不越界。

  • Router（路由层）：负责接收 HTTP 请求，解析参数，调用服务层获取数据，组装并返回响应。绝不包含业务逻辑。
  • Schema（数据层）：定义所有请求和响应的数据结构。使用 Pydantic 的 BaseModel，自带类型校验、默认值和序列化能力。
  • Service（业务层）：实现核心功能 — 文档解析、文本分块、向量嵌入、ChromaDB 操作、LLM 调用、RAG 管道编排。不依赖 HTTP 概念。

下面是一个典型的 Router 调用 Service、返回 Schema 的模式：

Python
# Router 调用 Service，返回 Schema — 标准三层协作模式
@router.post("/upload", response_model=DocumentUploadResponse)
async def upload_document(file: UploadFile = File(...)):
    content = await file.read()
    text = await parse_document(file.filename, content)  # Service 层
    chunks = chunk_text(text)                             # Service 层
    await store.add_document(...)                         # Service 层
    return DocumentUploadResponse(                        # Schema 层
        doc_id=doc_id,
        filename=file.filename,
        file_type=file_type,
        chunk_count=len(chunks),
        status="indexed"
    )
03

项目目录结构

整个 RAG 项目的目录结构如下，每个文件都有明确的职责分工：

目录树
main.py                    # FastAPI 应用入口 — CORS 配置、路由挂载、启动
app/
├── routers/                # 路由层：定义 API 端点
│   ├── __init__.py
│   ├── documents.py        # POST /upload  GET /  DELETE /{doc_id}
│   └── qa.py               # POST /ask  POST /compare (SSE 流式)
├── schemas/                # 数据层：Pydantic 模型定义
│   ├── __init__.py
│   ├── document.py          # DocumentUploadResponse, DocumentInfo,
│   │                        #   DocumentDeleteResponse, DocumentListResponse
│   └── qa.py                # QuestionRequest, AskResponse, CitationInfo,
│                            #   RetrievedChunkPreview, CompareResponse
├── services/               # 业务层：核心逻辑实现
│   ├── __init__.py
│   ├── llm.py               # AsyncOpenAI 客户端 — 封装 OpenAI API 调用
│   ├── embedding.py         # Embedding API 封装 — 文本向量化
│   ├── chunker.py           # 混合分块引擎 — RecursiveCharacterTextSplitter
│   ├── parser.py            # 文档解析 — PDF (pdfplumber), MD, TXT
│   ├── vector_store.py      # ChromaDB 操作 — add_document, search, delete
│   ├── rag_pipeline.py      # RAG 管道编排 — 检索→构建上下文→流式生成
│   └── streaming.py         # SSE 事件生成 — retrieval/answer/citations/done
└── utils/
    ├── __init__.py
    └── config.py            # 环境变量管理 — get_env() 辅助函数
设计原则：
  • 每个 Router 文件对应一组相关的 API 端点
  • 每个 Service 文件只做一件事（单一职责原则）
  • Schema 文件是 Router 和 Service 之间的数据契约
  • Service 层之间可以互相调用，但 Router 层不能直接访问外部 API
04

Pydantic Schema 设计

Pydantic Schema 是整个 API 的数据契约。它定义了「前端可以发送什么」和「前端可以期待收到什么」。来看看项目中的 Schema 设计：

文档相关 Schema (app/schemas/document.py)

Python
from pydantic import BaseModel, Field
from datetime import datetime

class DocumentUploadResponse(BaseModel):
    """文档上传成功后的响应"""
    doc_id: str
    filename: str
    file_type: str
    chunk_count: int
    status: str = "indexed"

class DocumentInfo(BaseModel):
    """文档列表中的单个条目"""
    doc_id: str
    filename: str
    file_type: str
    chunk_count: int
    created_at: datetime | None = None
    size_bytes: int = 0

class DocumentListResponse(BaseModel):
    """文档列表响应"""
    documents: list[DocumentInfo]
    total: int

class DocumentDeleteResponse(BaseModel):
    """文档删除响应"""
    doc_id: str
    deleted: bool = True
    message: str = "文档已删除"

QA 相关 Schema (app/schemas/qa.py)

Python
from pydantic import BaseModel, Field

class QuestionRequest(BaseModel):
    """提问请求体"""
    question: str = Field(..., min_length=1, max_length=5000,
                          description="用户问题")
    top_k: int = Field(default=5, ge=1, le=50,
                       description="检索结果数量")

class CitationInfo(BaseModel):
    """引用来源信息"""
    doc_id: str
    filename: str
    chunk_index: int
    content_snippet: str  # 前300字预览

class RetrievedChunkPreview(BaseModel):
    """检索到的 Chunk 预览"""
    doc_id: str
    filename: str
    chunk_index: int
    content_preview: str
    score: float

class AskResponse(BaseModel):
    """问答响应"""
    answer: str
    citations: list[CitationInfo]
    retrieved_chunks: list[RetrievedChunkPreview]
Pydantic 的优势：使用 Field() 可以定义字段的验证规则（长度、范围）和文档描述。FastAPI 会自动在 OpenAPI 文档中展示这些元信息。不合规的请求会被自动拒绝并返回 422 错误，无需任何手写验证逻辑。
05

路由挂载与中间件

FastAPI 应用的入口文件是 main.py，它负责创建应用实例、配置中间件、挂载路由。以下是核心代码：

Python — main.py
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware
from app.routers import documents, qa

app = FastAPI(
    title="个人知识库智能问答系统",
    version="0.1.0",
    description="基于 RAG 架构的个人文档智能问答 API"
)

# CORS 中间件 — 允许前端跨域访问
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)

# 挂载路由 — 统一的 /api/v1/ 前缀
app.include_router(documents.router, prefix="/api/v1/documents", tags=["文档管理"])
app.include_router(qa.router, prefix="/api/v1/qa", tags=["智能问答"])

@app.get("/")
async def root():
    return {"message": "个人知识库智能问答系统", "version": "0.1.0"}

@app.get("/health")
async def health_check():
    return {"status": "healthy"}

完整的 API 端点如下：

方法路径功能请求体
GET/根路径 — 返回服务信息无
GET/health健康检查无
POST/api/v1/documents/upload上传文档 — 解析→分块→索引multipart/form-data
GET/api/v1/documents获取已索引的文档列表无
DELETE/api/v1/documents/{doc_id}删除指定文档及其 Chunks无
POST/api/v1/qa/askRAG 问答 — 流式 SSE 响应QuestionRequest
POST/api/v1/qa/compareRAG vs 纯LLM 对比 — 交错 SSEQuestionRequest
06

错误处理

良好的错误处理让 API 使用者在遇到问题时能快速定位原因。FastAPI 提供了 HTTPException 来处理业务错误，Pydantic 提供自动的数据验证：

场景HTTP 状态码响应体处理方式
不支持的文件类型 400 {"detail": "不支持的文件类型: .exe"} Router 层检查扩展名，raise HTTPException
文件超出大小限制 413 {"detail": "文件大小超过限制 (10MB)"} 中间件或 Router 层检查 Content-Length
文档不存在 404 {"detail": "文档不存在: abc-123"} Service 层返回 None → Router 层抛异常
LLM API 错误 502 {"detail": "LLM服务暂时不可用"} 跨层异常处理器 (exception handler)
问题文本为空 422 {"detail": [{"msg": "ensure this value has at least 1 characters"}]} Pydantic 自动验证 — 无需手写代码
Embedding API 超时 504 {"detail": "Embedding服务请求超时"} Service 层捕获 TimeoutError → 抛 HTTPException
注意：Pydantic 的 Field(min_length=1, max_length=5000) 会自动拒绝空字符串和超长问题，返回 422 错误。这是声明式验证的强大之处 — 一行代码替代数十行手写验证逻辑。

📋

本课总结

  • FastAPI 是构建 RAG 系统 API 的最佳选择：异步、自动文档、Pydantic 集成、高性能
  • 三层架构（Router → Schema → Service）实现了关注点分离，每个层级职责清晰
  • Router 处理 HTTP 请求/响应，绝不包含业务逻辑
  • Schema 定义数据契约，Pydantic 自动完成验证和序列化
  • Service 封装核心业务 — LLM 调用、Embedding、Chunking、VectorStore
  • 项目目录结构按功能分层：routers/、schemas/、services/、utils/
  • CORS 中间件允许前端跨域访问，include_router 按路径前缀挂载路由
  • 错误处理结合 HTTPException 和 Pydantic 验证，提供清晰的错误响应
