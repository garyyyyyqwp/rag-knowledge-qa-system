向量相似度检索：余弦相似度与 Top-K

掌握余弦相似度的数学原理与 NumPy 实现，理解 Top-K 检索策略

01

为什么需要相似度计算？

将文本向量化以后，"找到与用户问题最相关的文档"这个任务就变成了一个向量空间中的最近邻搜索问题。我们需要一种方法来度量两个向量之间的"距离"或"相似度"。

常见的有三种度量方式：

余弦相似度（Cosine Similarity）

衡量两个向量方向的一致性。取值范围 [-1, 1]，1 表示完全相同方向，0 表示正交，-1 表示完全相反。

在文本检索中最常用——我们关心的是语义方向，而非文本长度。

欧氏距离（Euclidean Distance）

衡量两个向量在空间中直线距离。距离越小越相似。受向量长度影响较大，不适合直接用于文本相似度。

公式：d(A,B) = √[Σ(aᵢ - bᵢ)²]

点积（Dot Product）

两个向量对应分量相乘再求和。与余弦相似度密切相关，当向量归一化后，点积等于余弦相似度。

公式：A·B = Σ(aᵢ × bᵢ)

02

余弦相似度详解

余弦相似度是 RAG 系统中最重要的数学工具。它的公式如下：

取值范围解读：

余弦值 含义 示例（近似）
1.00 两个向量指向完全相同的方向 "猫" vs "猫"（完全相同文本）
0.85 ~ 0.95 语义高度相近 "今天天气很好" vs "今日气候宜人"
0.50 ~ 0.70 有一定关联但并非高度相似 "机器学习" vs "深度学习"
0.00 ~ 0.15 几乎没有语义关联 "今天天气很好" vs "我喜欢吃苹果"
-1.00 语义完全相反（在文本 Embedding 中极少出现） 仅在理论上存在
为什么用余弦相似度而非欧氏距离？ 余弦相似度关注的是方向而非绝对大小。一篇 1000 字的文章和一篇 100 字的摘要即使长度差异巨大，只要讨论的主题相同，它们的向量方向也应接近。欧氏距离会被长度差异误导，而余弦相似度不会。

NumPy 实现（三行代码）：

Python — NumPy 实现余弦相似度 
import numpy as np


def cosine_similarity(a: np.ndarray, b: np.ndarray) -> float:
    """计算两个向量的余弦相似度"""
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))


# 示例
a = np.array([1, 2, 3])
b = np.array([2, 4, 6])  # b 与 a 方向相同，只是更长
print(cosine_similarity(a, b))  # 输出 1.0 —— 方向完全相同！

c = np.array([1, 0, 0])
d = np.array([0, 1, 0])  # b 与 a 正交
print(cosine_similarity(c, d))  # 输出 0.0 —— 正交，完全无关

04

Top-K 检索策略

向量检索不是只返回 1 个最相关的结果，而是返回最相似的 K 个。K 的选择是一个重要的权衡：

K = 1（单一结果）

最精确但可能遗漏。如果第一个结果因为分块不当等问题恰好不包含完整答案，系统将无法回答。适合一些简单的精准匹配场景。

⚠ 不推荐：对复杂问题鲁棒性不足

K = 5（本项目默认值）

平衡精度与噪音的黄金选择。5 个 Chunk 通常能覆盖答案所需的多角度信息，又不会超出 LLM 的上下文窗口。

✅ 推荐：大多数 RAG 场景的最佳实践

K = 10（更多结果）

更全面但可能引入噪音。更多的 Chunk 可能包含不相关内容，稀释关键信息，增加 token 消耗和延迟。

⚠ 慎用：仅在确认检索质量高时增大 K

完整实现：

Python — Top-K 检索函数 
import numpy as np


def top_k_search(query_vec, doc_vecs, docs, k=5):
    """
    在文档向量中搜索与查询向量最相似的 Top-K 个文档
    
    参数:
        query_vec: 查询向量 (list[float])
        doc_vecs:  文档向量列表 (list[list[float]])
        docs:      原始文档列表 (list[str])
        k:         返回结果数量
    
    返回:
        [(文档内容, 相似度分数), ...]  按相似度从高到低排序
    """
    similarities = [
        np.dot(query_vec, dv) / (np.linalg.norm(query_vec) * np.linalg.norm(dv))
        for dv in doc_vecs
    ]
    # 获取相似度最高的 k 个索引（从高到低）
    top_indices = np.argsort(similarities)[-k:][::-1]
    return [(docs[i], float(similarities[i])) for i in top_indices]
项目中的配置 在 app/utils/config.py 中，RAG_TOP_K = 5 定义了默认检索数量。你可以根据实际问题类型动态调整：事实型问题用较小的 K（3~5），分析型问题用较大的 K（7~10）。
05

ChromaDB 中的相似度计算

在实际项目中，我们不会手动用 NumPy 计算相似度——ChromaDB 内置了高效的向量检索能力。但理解其底层原理对于调试和优化至关重要：

ChromaDB 使用余弦距离

ChromaDB Collection 的 metadata 中设置 hnsw:space: cosine，意味着它使用余弦距离作为相似度度量。

余弦距离 = 1 - 余弦相似度
距离越小 → 相似度越高

项目中转换为相似度分数

ChromDB 查询返回的是 distance。我们的项目将其转换回直观的相似度分数：

score = 1.0 - distance
分数越接近 1.0 → 越相关

Python — ChromaDB 检索示例 
# ChromaDB 查询返回包含 distance 的结果
results = collection.query(
    query_embeddings=[query_vector],
    n_results=5,          # Top-K = 5
    include=['documents', 'distances', 'metadatas']
)

# 将 distance 转换为可读的相似度分数
for doc, dist in zip(results['documents'][0], results['distances'][0]):
    score = 1.0 - dist
    print(f"相似度: {score:.4f} | 内容: {doc[:80]}...")

本课小结

  • 向量相似度计算是 RAG 检索的数学核心，用于在向量空间中找到与查询最接近的文档
  • 余弦相似度是最常用的度量：cos(θ) = (A·B) / (||A|| × ||B||)，取值范围 [-1, 1]
  • 语义相近的文本相似度可达 0.9+，语义无关的文本接近 0——差距非常明显
  • Top-K 策略决定检索返回的结果数量：K=5 是平衡精度与噪音的推荐默认值
  • ChromaDB 使用余弦距离（1 - 相似度），项目中将其转换为直观的相似度分数
