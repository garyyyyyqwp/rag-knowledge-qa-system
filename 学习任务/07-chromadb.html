第 7 课

ChromaDB 向量数据库实战

掌握 ChromaDB 的基本操作，实现向量持久化存储与高效检索

01

什么是向量数据库？

传统数据库

精确匹配查询

SELECT * FROM docs WHERE name = '张三'

查的是"等于"或"不等于"，结果非黑即白。

向量数据库

相似度查询

找到与 [0.1, 0.3, 0.5, ...] 最相似的 K 个向量

查的是"有多像"，返回 Top-K 最相关结果。

为什么不能直接用列表暴力搜索？

当数据量达到百万级时，对每条向量计算余弦相似度的复杂度是 O(N·D)（N = 数据量，D = 维度）。1536 维 × 100 万条 = 每次查询 15 亿次浮点运算——太慢了。

向量数据库使用近似最近邻（ANN）索引（如 HNSW），将复杂度降至 O(log N)，在牺牲可忽略精度的情况下实现毫秒级查询。

ChromaDB：轻量级、开源、Python 原生、0 配置即可启动。支持内存模式（EphemeralClient，进程结束即销毁）和持久化模式（PersistentClient，数据存磁盘）。

02

ChromaDB 核心概念

Client

数据库连接。分为 PersistentClient（持久化）和 EphemeralClient（临时），决定数据存在哪里。

Collection

相当于关系数据库的"表"。存储一组向量 + 文档 + 元数据。可以创建多个 Collection 做知识隔离。

Embedding

高维浮点数向量（本项目用 OpenAI text-embedding-3-small，1536 维）。向量的几何距离反映语义相似度。

Metadata

附加的结构化信息（文件名、类型、时间戳等）。用于过滤查询、标记来源和按条件删除。

Distance Function

相似度度量方式：cosine（余弦距离，默认推荐）、l2（欧氏距离）、ip（内积）。

基本操作实战

初始化与创建 Collection

Python 
import chromadb

client = chromadb.PersistentClient(path="./chroma_data")

collection = client.get_or_create_collection(
    name="knowledge_base",
    metadata={"hnsw:space": "cosine"}
)

插入向量

Python 
collection.add(
    ids=["doc1_0", "doc1_1"],
    embeddings=[[0.1, 0.2, ...], [0.3, 0.4, ...]],
    documents=["文本内容1", "文本内容2"],
    metadatas=[
        {
            "doc_id": "doc1",
            "filename": "notes.md",
            "chunk_index": 0
        },
        {
            "doc_id": "doc1",
            "filename": "notes.md",
            "chunk_index": 1
        },
    ]
)

相似度查询

Python 
results = collection.query(
    query_embeddings=[[0.15, 0.25, ...]],
    n_results=5,
    include=["documents", "metadatas", "distances"]
)

# results["documents"][0] → Top-5 文档文本
# results["metadatas"][0] → 对应元数据
# results["distances"][0]  → 余弦距离值

按元数据过滤删除

Python 
collection.delete(where={"doc_id": "doc1"})
# 删除所有 doc_id 为 "doc1" 的 Chunk

# 单例模式：全局唯一实例
_vector_store: VectorStore | None = None

def get_vector_store() -> VectorStore:
    global _vector_store
    if _vector_store is None:
        _vector_store = VectorStore(
            persist_dir=settings.CHROMA_PERSIST_DIR,
            collection_name="knowledge_base"
        )
    return _vector_store

单例模式：get_vector_store() 确保整个应用生命周期内只有一个 VectorStore 实例，避免重复创建 Collection 和多次加载索引。

05

Chunk 元数据设计

每条 Chunk 入库时都附带元数据，用于过滤、追溯和按文档删除：

字段类型说明
doc_id str 文档唯一标识（UUID hex），用于按文档过滤或删除
filename str 原始文件名（如 "论文笔记.pdf"）
file_type str 文件类型（pdf / md / txt），便于按类型过滤
chunk_index int Chunk 在文档中的序号（从 0 开始）
created_at str ISO 格式时间戳，记录入库时间
06

cosine 距离 → 相似度分数

ChromaDB 的 collection.query() 返回的是余弦距离（cosine distance），而非我们直观理解的"相似度分数"：

  • 余弦距离取值范围：[0, 2]（0 = 完全相同，2 = 完全相反）
  • 转换为相似度分数：score = 1.0 - distance
  • 相似度分数范围：[-1, 1]，但对于归一化向量，通常落在 [0, 1] 区间
Python 
# 项目中的距离转分数逻辑
for i in range(len(results["documents"][0])):
    distance = results["distances"][0][i]
    score = 1.0 - distance
    # score ≈ 1.0  → 高度相似
    # score ≈ 0.0  → 不相关
    # score ≈ -1.0 → 语义相反（极少出现）
07

持久化 vs 内存模式

PersistentClient

数据存磁盘，重启不丢失

配置：CHROMA_PERSIST_DIR=./chroma_data

✅ 生产环境标准选择
✅ 数据跨会话持久化
✅ 目录下生成 SQLite + 索引文件

EphemeralClient（测试用）

进程结束即销毁，每次运行都是全新状态

创建：chromadb.EphemeralClient()

✅ 适合 pytest 单元测试
✅ 无需清理残留数据
⚠️ 数据不跨会话保留

本项目使用 PersistentClient。CHROMA_PERSIST_DIR 通过 .env 文件可配置，默认路径为 ./chroma_data。

本课要点

  • 向量数据库做相似度查询，与传统关系数据库的精确匹配查询有本质区别
  • ANN 索引（HNSW）将百万级向量搜索从 O(N) 降至 O(log N)
  • ChromaDB 核心概念：Client → Collection → Embedding / Document / Metadata
  • 5 大基本操作：创建 Collection → 添加向量 → 查询 → 删除 → 列出文档
  • 项目用 VectorStore 封装 ChromaDB，单例模式确保全局唯一实例
  • 元数据设计（doc_id / filename / file_type / chunk_index / created_at）支持精确过滤和追溯
  • 余弦距离转换公式：score = 1.0 - distance，分数越接近 1.0 越相似
  • PersistentClient 用于生产，EphemeralClient 用于测试
